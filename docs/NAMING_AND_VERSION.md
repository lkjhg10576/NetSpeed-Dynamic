# Music Dynamic Island（MDI）命名规范与版本号规则

> 本文档统一规定 MDI 项目中**界面/点位的中文标准命名**与**版本号规则**，
> 用于消除代码、UI、文档三处命名不一致的问题。
> 适用范围：所有新增与修改代码、UI 文案、注释、文档。

---

## 1. 核心概念术语表

| 术语 | 含义 |
|------|------|
| 控制台 | 主窗口（Tauri label=`main`），承载全部设置，代码 `MainPanel.vue` |
| 灵动岛 | 浮窗窗口（Tauri label=`widget`），真正的动态岛本体，代码 `WidgetIsland.vue` |
| 实时活动枢纽 | `LiveActive.vue`，番茄钟/倒计时/硬件监控/健康提醒/动态感知的统一配置中枢 |
| 主岛区 | 灵动岛左侧区域，含网速状态与音乐控制 |
| 实时活动区 | 灵动岛右侧区域，实时活动入口（点击展开） |
| 点位 | 灵动岛在屏幕上的停靠位置，由用户手动拖动决定，分两种模式（见 §3.6） |

## 2. 界面 / 点位命名

### 2.1 窗口（Tauri Window）

| label | 旧称（混乱点） | 标准名 | 代码/配置 | 说明 |
|-------|---------------|--------|-----------|------|
| `main` | MDI 控制台 / 设置 / 面板 | **控制台** | `MainPanel.vue`、`tauri.conf.json` | 设置主窗口，标题应统一为「MDI 控制台」或「控制台」 |
| `widget` | MDI Widget / 灵动岛 | **灵动岛** | `WidgetIsland.vue`、`tauri.conf.json` | 浮窗本体，标题应统一为「灵动岛」 |

### 2.2 视图（View / 内嵌面板）

| 文件 | 旧称（错误） | 标准名 | 说明 |
|------|-------------|--------|------|
| `LiveActive.vue` | 灵动岛设置（误称） | **实时活动枢纽** | 它是实时活动的配置中枢，不是「灵动岛」的设置；`MainPanel.vue` 中 `currentView==='island'` 指向它，标题误写「灵动岛设置」需改为「实时活动枢纽」 |

### 2.3 灵动岛内部区域

| 区域 | 标准名 | 内容 |
|------|--------|------|
| 左侧 | **主岛区** | 网速状态（good/warning/error）+ 音乐控制 |
| 右侧 | **实时活动区** | 实时活动小图标（rt-chip），点击展开对应活动 |

> 旧代码用「左胶囊 / 右圆」等形态化描述，文档与注释统一改用「主岛区 / 实时活动区」。

### 2.4 实时活动（5 种，代码 id → 标准中文名）

| 代码 id | 标准中文名 | 说明 |
|---------|-----------|------|
| `pomodoro` | **番茄工作法** | 番茄钟 |
| `countdown` | **倒计时** | 倒计时 |
| `hardware` | **硬件监控** | CPU/内存等硬件状态 |
| `health` | **健康提醒** | 含「久坐」+「喝水」两条子提醒 |
| `sysmsg` | **动态感知** | 系统动态感知（不参与轮换，常驻） |

> 代码内部 `id` 保留英文（`pomodoro` 等）作为内部标识；**所有面向用户的 UI 文案使用上表中文标准名**。

### 2.5 控制台设置分组（保持现状）

以下分组名**维持不变**，仅作集中登记，避免后续被改名造成混乱：

1. 常规设置
2. 音乐控制平台（六宫格）
3. 音乐控制器 PRO
4. 硬件监控模式
5. 自动隐藏 / 折叠
6. 材质（acrylic / mica / blur / none）
7. 开机自启
8. 省内存模式

### 2.6 灵动岛停靠模式（点位）

灵动岛位置由用户手动拖动决定，分两种模式：

| 旧描述 | 标准名 | 说明 |
|--------|--------|------|
| 不在任务栏 | **自由模式** | 灵动岛悬浮于桌面任意位置，不吸附任务栏 |
| 在任务栏 | **任务栏吸附模式** | 灵动岛吸附于任务栏上方 |

> 旧代码键 `NSD_ISLAND_POSITION` 仅存用户拖动后的坐标，两种模式由是否吸附任务栏区分。

## 3. 版本号规则

### 3.1 格式

```
v主版本.次版本.修订-序号
```

示例：`v1.2.3-4` 表示主版本 1、次版本 2、修订 3 的**第 4 个测试版**。

### 3.2 各段含义

| 段 | 名称 | 含义 |
|----|------|------|
| `主版本` | 大版本 | 巨大变动：架构变动、独特大功能加入、视觉体验重构等 |
| `次版本` | 普通版本 | 普通小功能加入等「有变动但不大」的情况 |
| `修订` | 例行版本 | 对应当前次版本的 bug 修复、体验优化等例行变动 |
| `序号` | 测试版号 | 项目**不区分内测 / 公测**，只有「测试」与「正式」之分；`-序号` = 该版本的第几个测试版（如 beta1 / beta2 / beta3） |

### 3.3 测试版 vs 正式版

- **测试版**：带 `-序号`。例如 `v1.2.3-1`、`v1.2.3-2` …… 即同一 `主.次.修订` 先经历若干测试版。
- **正式版**：去掉 `-序号`，仅 `v主.次.修订`。例如 `v1.2.3`。
- 逻辑：同一 `主.次.修订` 先发若干测试版（`-1/-2/-3…`），最终发布无后缀的正式版。

### 3.4 升版规则（何时 +1）

| 变动类型 | 操作 | 其余段处理 |
|----------|------|-----------|
| 架构变动 / 独特大功能 / 视觉重构 | **主版本 +1** | 次版本、修订归 0，序号从 1 起 |
| 新增普通小功能 | **次版本 +1** | 修订归 0，序号从 1 起 |
| bug 修复 / 体验优化（例行） | **修订 +1** | 序号从 1 起 |
| 发布该版本的下一个测试包 | **序号 +1** | 主/次/修订不变 |

> 规则要点：**序号随「主/次/修订」任一变化而重置为 1**；序号只在同一版本的三段不变时累加，代表第几个测试包。

### 3.5 版本号同步（三处必须一致）

版本号出现在三处，发布前必须手动核对**完全一致**（本项目采用手动同改，无自动同步脚本）：

| 文件 | 字段 | 当前值 |
|------|------|--------|
| `package.json` | `version` | `0.4.0-2` |
| `src-tauri/tauri.conf.json` | `version` | `0.4.0-2` |
| `src-tauri/Cargo.toml` | `[package] version` | `0.4.0-2` |

当前版本解读：`v0.4.0-2` = 0.4.0 的**第 2 个测试版**（0.4.0 测试尚未结束，不升次/修订位）。

## 5. 命名总则

1. **中文显示名优先**：所有面向用户的 UI 文案、文档使用中文标准名；英文仅作代码内部 `id` / 变量标识。
2. **一词一义**：同一概念在全项目只能用同一个标准名，禁止同物异名（如「控制台 / 设置 / 面板」只能叫「控制台」）。
3. **三处一致**：UI 文案、代码注释、文档须统一使用本标准名。
4. **版本号三处一致**：`package.json` / `tauri.conf.json` / `Cargo.toml` 的版本号发布前必须核对一致。
